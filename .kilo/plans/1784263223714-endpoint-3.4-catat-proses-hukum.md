# Plan: Endpoint 3.4 — Catat Riwayat Proses Hukum

## Context

CodeIgniter 3 API. ERD v1.0: role_id 1=Super Admin, 2=Operator Polda, 3=Eksekutif.

Two critical architectural bugs found:
1. DB seed v5 has **wrong labels**: id=2 'Superadmin', id=3 'Operator Polda' (inverted).
2. Auth.php JWT payload uses key `'role'` instead of `'role_id'` and omits `polda_id` — breaking all decentralized yurisdiksi gates.

**Bug 2 is live in production**: every controller reads `$payload['role_id']` but Auth.php writes `$payload['role']`. This means **every role check currently evaluates to 0** and every yurisdiksi gate sees `polda_id=0`. The entire auth/decentralization layer is non-functional until Auth.php is patched.

**Cascading damage**: After fixing Auth.php to emit correct `role_id` and `polda_id`, 4 controllers have hardcoded role checks against the wrong IDs (they were coded against the DB seed's inverted mapping). All must be patched atomically.

## Phase 1: Create `tbl_proses_hukum` Table

File: `database/v5/sindomondb.sql`

Add before COMMIT:
```sql
CREATE TABLE IF NOT EXISTS `tbl_proses_hukum` (
  `hukum_id` INT(11) NOT NULL AUTO_INCREMENT,
  `personil_id` VARCHAR(36) COLLATE utf8mb4_unicode_ci NOT NULL,
  `klasifikasi` ENUM('Pemeriksaan Propam','Sidang Kode Etik','Sidang Disiplin','Pidana Umum') COLLATE utf8mb4_unicode_ci NOT NULL,
  `status_hukum` VARCHAR(100) COLLATE utf8mb4_unicode_ci NOT NULL,
  `tanggal_mulai` DATE NOT NULL,
  `deskripsi_kasus` TEXT COLLATE utf8mb4_unicode_ci NULL,
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`hukum_id`),
  INDEX `idx_personil_id` (`personil_id`),
  CONSTRAINT `fk_proses_hukum_personil` FOREIGN KEY (`personil_id`)
    REFERENCES `tbl_personil`(`personil_id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

**Prerequisite**: `tbl_personil` must use InnoDB. If MyISAM, add:
```sql
ALTER TABLE tbl_personil ENGINE=InnoDB;
```

## Phase 3: Controller — `hukum_post()` Method

File: `application/controllers/Sdm.php`

Add new method after `personil_put()` (after line 458), before `_build_tree`.

Flow:
1. **AUTH**: `get_jwt_payload($this)`. If null → 401.
2. **ROLE GATE**: `$role_id != 2` → 403 (Operator Polda only).
3. **EXTRACT**: `$jwt_polda_id` from payload.
4. **PARSE JSON**: `$this->input->raw_input_stream`. If empty → 400.
5. **VALIDATE mandatory**: `personil_id, klasifikasi, status_hukum, tanggal_mulai`. If any missing → 422.
6. **ENUM VALIDATE**: `klasifikasi` must be one of: `['Pemeriksaan Propam', 'Sidang Kode Etik', 'Sidang Disiplin', 'Pidana Umum']`. If invalid → 400.
7. **CROSS-TABLE VERIFY**:
   ```sql
   SELECT polda_id FROM tbl_personil WHERE personil_id = ?
   ```
   If no row OR `polda_id != $jwt_polda_id` → 403 "Akses ditolak. Personel tidak ditemukan atau berada di luar yurisdiksi Anda."
8. **INSERT**:
   ```sql
   INSERT INTO tbl_proses_hukum (personil_id, klasifikasi, status_hukum, tanggal_mulai, deskripsi_kasus)
   VALUES (?, ?, ?, ?, ?)
   ```
9. **SUCCESS**: HTTP 201 `{"status":201,"message":"Catatan hukum berhasil ditambahkan.","data":new stdClass()}`.

Follow existing Sdm.php response envelope pattern:
```php
$this->output->set_status_header(XXX);
echo json_encode(array(
    "message" => "...",
    "status" => XXX,
    "data" => new stdClass()
));
```

## Deployment Order

All changes must be deployed **atomically** in a single PR. Deploying Phase 0 without Phases 0d-0f would leave Kamtibmas, Dms, and Pengaduan with broken role checks for 3 turns (until their patches arrive).

Order:
1. Apply SQL migrations (0a, 0b, 1) in one transaction.
2. Deploy all PHP file patches (0c, 0d, 0e, 0f, 2, 3) together.
3. All existing JWT tokens invalidated — users must re-login.
4. Run curl tests (Phase 4).

## Risks

1. **tbl_personil engine**: FK requires InnoDB. If MyISAM, migration fails silently unless ALTER is included.
2. **Existing tokens**: All current JWTs use key `'role'` and lack `polda_id`. After 0c, they are rejected. Users re-login. Acceptable in dev.
3. **tbl_users.polda_id population**: No automated migration for existing rows. Operator users need manual UPDATE after ALTER.
